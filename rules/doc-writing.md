# 문서 작성 규칙 (Doc Writing Rules)

Claude가 이 저장소에서 문서를 작성하거나 수정할 때 반드시 따라야 할 규칙입니다.

---

## 1. 언어 및 문체

- **기본 언어**: 한국어
- **영어 원문 병기**: 기술 용어는 첫 등장 시 반드시 영어 원문 병기
  - 예: 네트워크 정책 (CiliumNetworkPolicy), eBPF, 엔드포인트 (Endpoint), 허블 (Hubble)
- **문체**: 명사형 종결 (`~입니다` 대신 `~함`, `~사용`)
- **존댓말 금지**: 기술 문서 특성상 경어체 사용하지 않음

## 2. 코드 블록

- **언어 태그 필수**: 모든 코드 블록에 언어 지정
  ```bash   # kubectl / cilium / hubble 명령어
  ```yaml   # Kubernetes / Cilium CRD YAML
  ```json   # JSON 설정
- **실행 가능한 코드**: 복붙해서 바로 적용되는 수준으로 작성
- **플레이스홀더 표시**: 교체 필요한 값은 `<YOUR_VALUE>` 형식으로 명시

## 3. 섹션 구성 필수 요소

새 문서는 반드시 아래 5개 섹션을 포함해야 함:

| 섹션 | 내용 | 생략 가능 여부 |
|------|------|----------------|
| 1. 개요 | 무엇인지, 왜 필요한지 | 불가 |
| 2. 설명 | 원리, YAML 코드, Best Practice | 불가 |
| 3. 트러블슈팅 | 증상 → 원인 → 해결 | 불가 |
| 4. 모니터링 및 확인 | cilium CLI / Hubble / Prometheus 확인 방법 | 불가 |
| 5. TIP | 현장 팁, 관련 링크 | 가능 |

## 4. 표 작성 규칙

- 비교 정보는 반드시 표(table)로 정리
- 3개 이상의 항목을 나열할 때는 불릿 대신 표 사용

## 5. 트러블슈팅 작성 형식

```markdown
### 증상
- 어떤 오류 메시지가 나타나는지 정확히 기술

### 원인
- 왜 발생하는지 근본 원인 설명

### 해결 방법
```bash
# 진단 명령어
cilium status
hubble observe --namespace <NAMESPACE>

# 해결 명령어
kubectl apply -f <FIX_YAML> -n <NAMESPACE>
```
```

## 6. 금지 사항

- 추측성 내용 작성 금지 (실제 경험 기반만)
- "~할 수 있습니다", "~일 것입니다" 같은 불확실한 표현 금지
- 공식 문서 링크 없이 수치(타임아웃 권장값, 한도) 단독 기재 금지
